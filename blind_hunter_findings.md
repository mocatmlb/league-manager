### Blind Hunter Review (Diff Only)

#### Findings:

1. **Inconsistent Request Status Mapping in `RescheduleService::cancel()`**: The service maps a coach's cancellation to `Denied` status in the database. This is semantically confusing. If the coach cancels it, it should ideally be `Cancelled` or a similar status, but the code comments say the ENUM only supports `Pending`, `Approved`, `Denied`. Using `Denied` for a user-initiated cancellation might trigger "Request Denied" emails or logic that isn't appropriate for a self-cancellation.
2. **Missing Input Validation in `RescheduleService::submit()`**: The method accepts `$requestData` and accesses keys like `requested_date`, `requested_time`, etc., without checking if they exist. This will cause PHP warnings (or Errors in PHP 8+) if the array keys are missing.
3. **Potential SQL Injection in `RescheduleService::getEligibleGames()`**: The `$placeholders` are generated using `implode(',', array_fill(0, count($teamIds), '?'))`, which is good. However, if `$teamIds` is empty, it returns `[]` early, which is safe. But waitâ€”is `$teamIds` guaranteed to be an array of integers? `TeamScope::getScopedTeams` is external, so we should trust but verify or cast.
4. **`RescheduleService::submit()` Notification try-catch is too broad?**: It catches `Throwable`, which is safe, but it logs the error and continues. If the notification system is down, the user gets no feedback, but the request is saved. This is acceptable per comments but might be worth noting.
5. **No Validation on `requested_date` format in `submit()`**: The service takes the string directly from `$requestData['requested_date']`. If it's not a valid date, the DB might reject it or store a zero-date, depending on MySQL mode.
6. **`RescheduleService::cancel()` Race Condition**: It fetches the request, checks the status, then updates. Two concurrent cancel requests could both pass the `Pending` check and then both update to `Denied`. While the result is the same, if there were side effects (like logging or notifications), they would double-fire.
7. **Type Inconsistency in `schedule-change.php`**: In the POST handler, `game_id` is cast to `(int)`, but `request_id` for cancel is also cast to `(int)`. This is fine, but the service methods take `int`, so it's consistent. However, the `request_id` from `$_POST` should be checked for existence before casting.
8. **Hardcoded URLs in `schedule-change.php`**: Redirects use `header('Location: schedule-change.php')`. If the site is in a subdirectory or has specific routing, this might break, although it's usually fine for relative paths in PHP.
9. **Missing `exit()` after `http_response_code(403)`**: In `schedule-change.php`, when a `TeamScopeViolationException` is caught, it sets 403 and an error message, but then continues to render the whole page. Usually, a 403 should stop execution or show a dedicated error page.
10. **`RescheduleService::getCoachRequests()` joins `schedules` but use `LEFT JOIN`?**: It uses `JOIN schedules s ON g.game_id = s.game_id`. If a game doesn't have a schedule entry yet (rare but possible), this request will vanish from the coach's list. `getEligibleGames` explicitly filters out NULL game_date, but the request list shouldn't hide requests for unscheduled games if they were somehow allowed to be created.
